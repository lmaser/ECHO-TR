ECHO-TR JIT rework plan

Decision: hay que rehacer la arquitectura de JIT. No hace falta tratar divisiones concretas ni crear mas tramos. El fallo principal era que el jitter calculado se estaba metiendo antes del smoothing general del delay.

Estado: la separacion topologica ya esta aplicada en las rutas Mono, Stereo, PingPong, Wide, Dual y Reverse. El smoothing largo sigue afectando al delay base; JIT se aplica despues para la lectura.

1. Hallazgo estructural

El codigo actual calcula un delay con jitter:

targetDelayNow = targetDelay * getJitterDelayMultiplier(...)
smoothedDelaySamples = smoothedDelaySamples * smoothCoeff
                     + targetDelayNow * (1 - smoothCoeff)
readPos = writePos - smoothedDelaySamples

Pero smoothCoeff viene de kDelaySmoothTauSeconds = 0.08. Eso equivale a un low-pass de unos 2 Hz. Por tanto, aunque el motor calcule flutter a cientos o miles de Hz, la ruta DSP lo aplasta antes de leer la linea de delay.

3. Arquitectura correcta

Separar base delay smoothing de jitter modulation.

// 1) El parametro TIME/SYNC/MIDI sigue usando smoothing largo anti-zipper.
baseSmoothedDelay = smooth80ms(targetDelay)

// 2) JIT se calcula sample-accurate y se aplica despues del smoothing base.
jitterMultiplier = exp2(jitterOut * jitterDepthOct)
jitteredDelay = max(2, baseSmoothedDelay * jitterMultiplier)

// 3) La lectura usa jitteredDelay directamente.
readPos = writePos - jitteredDelay

Esto mantiene suaves los cambios de parametro, pero deja que JIT tenga velocidad real. No es un fix por tramos: es corregir la topologia.

4. Modelo matematico propuesto

La profundidad de JIT debe ser una proporcion temporal del delay. No debe decrecer con TIME.

a  = clamp(JIT / 100)
aP = a * a

targetDepthRatio = 0.055 * pow(aP, 1.05)
depthSeconds = baseDelaySeconds * targetDepthRatio
depthOct = log2((baseDelaySeconds + depthSeconds) / baseDelaySeconds)

El tiempo decide el reparto perceptual, no la intensidad:

short time -> mas rate, mas tone layer, mas caracter timbrico
long time  -> menos rate, mas recorrido absoluto en ms, terremoto lento

5. Curva de velocidad despues del cambio de arquitectura

Una vez JIT deje de pasar por el smoother de 80 ms, la curva de velocidad debe revisarse desde cero. El exponente 0.90 puede ser demasiado rapido en cortos cuando ya no este filtrado. La primera version segura debe mantener la ley continua, pero con limites musicales.

flutterRateHz = (4 + 126 * aP) * pow(250 / delayMs, exponent)
exponent inicial recomendado: 0.75 - 0.85 despues de quitar el smoother de 80 ms

Importante: esto no es tratar sectores. Es recalibrar una unica ley global tras quitar un filtro que estaba invalidando todos los calculos anteriores.

6. Plan por fases

  1. Fase 1: aplicada. JIT va despues del smoothing base en Mono/Stereo/PingPong/Wide/Dual.
  2. Fase 2: aplicada. Reverse usa delay base suavizado y jitter posterior para chunk/lecturas derivadas.
  3. Fase 3: pendiente de escucha. Recalibrar una sola curva global de velocidad solo si el motor directo queda excesivo o corto.
  4. Fase 4: pruebas A/B en 5 ms, 10 ms, 1/16, 1/8, 500 ms, 1/1 y 2/1 con JIT 25/50/100.
  5. Fase 5: retirar formulas/documentacion antiguas que hablen de JIT antes del smoother.

7. Criterio de aceptacion

  • 5 ms: debe fluctuar rapido de verdad, no como LFO lento.
  • 1/16 y 1/8: debe ser evidente en JIT 50 y creativo en 100.
  • 500 ms o mas: debe sentirse como movimiento lento grande, no como ausencia de efecto.
  • Sin tramos especiales: si hace falta tocar divisiones concretas, el modelo sigue mal.
