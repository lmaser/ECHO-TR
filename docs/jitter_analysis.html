ECHO-TR JIT analysis

Estado anterior, calculos y modelo de reimplementacion aplicado para que JIT tenga intensidad perceptual coherente en todo el rango de TIME/SYNC/MOD.

Conclusion corta: el motor actual no esta roto por una sola constante. Esta mezclando varias ventanas por tiempo, una profundidad dependiente de delay y un S&H suavizado cuyo efecto perceptual depende del producto rate * depth_seconds. Por eso tocar "mas rapido" varias veces no ha compensado bien 1/16, 1/8 y 1/1.

1. Ruta actual de parametros

TIME/SYNC

Primero se obtiene targetDelayMs desde MIDI, Sync o Time manual. En Sync se usa una tabla de divisiones en notas negras.

MOD

Luego se calcula freqMultiplier. El delay final para JIT es targetDelayMs / freqMultiplier. Esto significa que JIT si ve el tiempo final despues de MOD.

JIT

JIT modula el delay como multiplicador logaritmico: delay * exp2(randomOut * depthOct), y tambien modula feedback.

targetDelayMs = MIDI ? 1000/freq : Sync ? tempoSyncToMs(index,bpm) : timeMs
freqMultiplier = mod < 0.5 ? 1/(4 - 6*mod) : 1 + (mod - 0.5)*6
delaySamples = sampleRate * targetDelayMs / 1000 / freqMultiplier
jitAmountMapped = JIT * JIT
delayMultiplier = exp2(jitterDelayOut * depthOct)

4. Problemas detectados

Problema A: rango util comprimido arriba

Se ha mejorado con amtMapped, pero la arquitectura sigue teniendo submotores con thresholds: fastMix, toneAmount, high y extreme. Eso puede hacer que bajar un poco el knob quite capas enteras de movimiento.

Problema B: tiempo corto no equivale a efecto fuerte

En delays cortos, la profundidad absoluta en ms es pequena. Subir solo la velocidad no basta si el producto rate * depth_seconds queda por debajo del de tiempos largos.

Problema C: ventanas de tiempo por escalones

Las ventanas highFreqDelay, shortDelay, veryShortDelay y longDelay son continuas matematicamente, pero crean regimenes perceptuales distintos. Al cruzar una zona, puede parecer discontinuo.

Problema D: un solo motor intenta hacerlo todo

El mismo JIT intenta cubrir drift lento, flutter, S&H irregular y pseudo-ring/tone. Deberian ser capas separadas con pesos normalizados por tiempo.

6. Modelo aplicado desde cero

Objetivo: JIT 50% debe ser claramente audible en 1/16 y 1/8 sin depender de estar al 70-100%. JIT 100% debe ser creativo y casi excesivo, pero usable. La percepcion debe ser similar entre tiempos cortos y largos.

Modelo propuesto

// 1) Amount perceptual, una sola curva para todos los submotores
a = clamp(JIT / 100)
aP = a * a                       // curva UX global: el 50% nuevo equivale aprox. al 20% anterior

// 2) Tiempo en escala logaritmica, no por ventanas duras
Dms = max(1000 * delaySamples / sr, 0.05)
shortness = clamp(log2(500 / Dms) / log2(500 / 8), 0, 1)
longness  = clamp(log2(Dms / 250) / log2(4000 / 250), 0, 1)

// 3) Capas separadas
driftRateHz   = mix(0.08, 1.2, aP) * mix(1.0, 0.35, longness)
flutterRateHz = mix(4.0, 130.0, aP) * pow(250 / Dms, 0.90)
flutterRateHz = clamp(flutterRateHz, 2.0, 7000.0)

// 4) Capa tone/phase-noise ligada a la frecuencia del delay
delayHz = 1000 / Dms
toneRateHz = delayHz * (4.0 + 190.0*aP + high*60.0 + extreme*90.0) * pow(shortness, 0.70)
toneRateHz = clamp(toneRateHz, 0.0, sr * 0.22)

// 5) Profundidad temporal de JIT: misma intensidad relativa, distinto reparto perceptual
// JIT no es variation. El recorrido debe ser apreciable y no decrecer en tiempos largos.
targetDepthRatio = 0.055 * pow(aP, 1.05)
depthSeconds = Dms * 0.001 * targetDepthRatio
depthSeconds = clamp(depthSeconds, 0.0000001, Dms * 0.001 * 0.12)
depthOct = log2((Dms*0.001 + depthSeconds) / (Dms*0.001))

Arquitectura de capas

  • Drift: lento, aleatorio y suavizado. Debe dominar en tiempos largos.
  • Flutter S&H: rapido, con slew/lag controlado. Debe estar presente desde JIT medio.
  • Tone layer: oscilador irregular ligado a delayHz, para que en tiempos cortos haya efecto timbrico/ring-like real.
  • Depth law: calcular profundidad como proporcion temporal del delay. En tiempos cortos hay mas velocidad/tone; en tiempos largos mas recorrido en ms.
  • No thresholds duros: usar curvas logaritmicas suaves para evitar cambios de regimen en 1/8, 1/16, etc.

7. Plan de implementacion recomendado

Fase 0: congelar estado

No tocar mas constantes. Mantener este HTML como referencia de calculo y comparar antes/despues.

Fase 1: funcion comun de metricas

Crear una funcion interna que, dado delaySamples, JIT y sr, devuelva shortness, longness, delayHz y targetIndex.

Fase 2: reemplazar rates

Sustituir las ventanas actuales por driftRateHz, flutterRateHz y toneRateHz normalizados. No tocar feedback ni engines.

Fase 3: normalizar profundidad

Calcular depthOct desde targetIndex. Esto deberia igualar 1/16, 1/8 y 1/1 a nivel perceptual.

Fase 4: pruebas A/B

Probar Sync 1/16, 1/8, 1/4, 1/1 a BPM 130 con JIT 25/50/75/100. La progresion debe ser monotona y compensada.

Fase 5: retirar legacy

Cuando este validado, eliminar constantes antiguas y documentar la nueva semantica de JIT.
